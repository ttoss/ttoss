Abaixo está uma **matriz formal de benchmark comparativo de paridade de mercado**. Ela foi desenhada para responder à pergunta: **“independentemente do diferencial semântico do ttoss, essa library compete bem no baseline que o mercado espera?”** Esse baseline aparece de forma consistente nas referências oficiais: APG/WAI-ARIA para acessibilidade e interação, Storybook para documentação e testes, DTCG e Open UI para tokens/customização, MUI para plataformas suportadas e disciplina de bundle, e políticas públicas de release para confiança operacional.

Use sempre a mesma coorte de comparação. Para o ttoss, a coorte principal deve ser **Ark UI, Radix, React Aria, Headless UI e Base UI**, porque todas se posicionam como foundations/headless ou low-level accessible UI; se quiser um controle adicional “mais opinionated”, adicione **MUI** em uma coluna separada. ([Ark UI][2])

## Como usar esta matriz

Use a mesma bateria de tarefas, o mesmo app de teste, a mesma matriz de browsers e o mesmo conjunto de stories para todas as libraries. Dê uma nota **0 a 5** para cada linha, aplique o peso, e some no final. Fórmula: **score ponderado = (nota ÷ 5) × peso**.
Escala recomendada:

- **0** = inexistente ou inadequado
- **1** = muito fraco
- **2** = abaixo do mercado
- **3** = paridade aceitável
- **4** = forte
- **5** = excelente / referência

Também recomendo 3 gates eliminatórios:

- **Acessibilidade < 3** = não entra como library madura
- **Compatibilidade < 3** = não entra como library segura para produção
- **Manutenção/Releases < 3** = não entra como library confiável no tempo

## Instruções curtas de execução

1. **Fixe uma bateria única de tarefas** para todas as libraries. Não troque o escopo entre concorrentes.
2. **Fixe uma bateria única de stories/tests**. O ideal é usar Storybook como harness comum para a maior parte da avaliação. Storybook foi desenhado exatamente para desenvolvimento, documentação e testes de UI em isolamento.
3. **Colete evidência objetiva**, não impressão: relatório de a11y, bundle report, release log, vídeos curtos de keyboard flow, links de docs, e tempo real de implementação.
4. **Atribua nota com âncora**. Nunca use “4 porque parece bom”; use “4 porque passou 85% dos patterns com zero workaround”, por exemplo.
5. **Calcule o score ponderado** e mantenha separado do score estratégico do ttoss.

## Regras para manter a comparação honesta

Não dê bônus ao ttoss nesta matriz por causa do seu diferencial semântico. Aqui ele precisa competir nos pontos de paridade. O diferencial semântico deve entrar **num segundo score**, separado.
Também não misture libraries de natureza diferente sem marcar a coorte. Ark UI, Radix, React Aria, Headless UI e Base UI formam uma coorte mais justa de comparação porque todas se apresentam como bases acessíveis/headless ou low-level para design systems; MUI é melhor como controle adicional mais opinionated.
